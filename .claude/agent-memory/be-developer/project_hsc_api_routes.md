`app/src/app/api/**/route.ts` implements ADR-000 §6. First route committed at `554bcd6` on branch `moja-praca` (`POST /api/cases`). See [[project_hsc_ai_layer]], [[project_hsc_persistence_layer]], [[project_hsc_image_storage]], [[project_hsc_policies_loader]] for the lib layers routes call.

**DI seam pattern (reuse for every route):** export `createXHandler(deps)` that returns the HTTP-method function, plus a thin exported `POST`/`GET` that wires production defaults (`getDb()`, `createModels()`). Integration tests import the factory and inject a temp SQLite handle (`createDb(tempPath)`), temp `uploadsBaseDir`, temp `policiesDir`, and mock models — zero network, zero env. `deps` for `POST /api/cases`: `{ db, models: Models, uploadsBaseDir?, policiesDir? }`.

**CRITICAL test-env gotcha:** route integration tests that call `request.formData()` (or otherwise exercise undici Request/Response body parsing) MUST start the file with `// @vitest-environment node`. The project's vitest default environment is `jsdom` (for React tests), and jsdom's `Request.formData()` throws on a multipart body — every request silently fell into the route's parse-error 400 branch until the directive was added. Build a real `Request` with a `FormData` body (Node 24 has global `File`/`FormData`/`Request`); it round-trips correctly under the node env.

**`POST /api/cases` request/response shapes as implemented (P3.1 consumes these):**
- Request: `multipart/form-data`. Create: `requestType` (`zwrot`|`reklamacja`), `category`, `productName`, `purchaseDate` (ISO), `description` (required iff reklamacja), `image` (File, JPG/PNG/WebP ≤10MB). Retry: a non-empty `caseId` field and NO image → re-runs AI pipeline only against the stored image (fresh ImageAnalysis+Decision, no new Case/CaseImage rows).
- 200 conclusive: `{ caseId, caseNumber, decision: { status: "approved"|"rejected"|"needs_human_review", justification, nextSteps: string[], disclaimer }, requiresBetterPhoto: false }`. `status` is the raw enum (frontend maps to Polish via `pl.chat.decisionLabels`); `disclaimer` is `pl.chat.disclaimer`.
- 200 inconclusive first analysis: `{ caseId, caseNumber, decision: null, requiresBetterPhoto: true }` — NO decision row; a first assistant chat message asking for a better photo is persisted.
- 400 validation: `{ fieldErrors: Record<field, polishMessage> }` (messages straight from `pl.form.errors`); ZERO AI calls before this.
- 404 (retry unknown caseId): `{ error: pl.errors.caseNotFound }`.
- 502 AI failure: `{ retryable: true, error, caseId, caseNumber }`. **Deviation from ADR's literal `{ retryable: true }`:** `caseId`/`caseNumber` are added so the client can actually retry the AI-only path (case+image are already persisted before the LLM calls). Load-bearing field is still `retryable: true`.

**AC-35 (persistence-failure isolation):** the first assistant chat-message insert is treated as NON-critical — wrapped in try/catch, logged via `console.error`, decision still returned 200. Decision/analysis/case/image inserts are critical (not wrapped).

**mime_type stored for case_images is always `"image/jpeg"`** (compressImage re-encodes to JPEG); `original_filename` keeps the uploaded name. On retry, `lastImage.mimeType` (typed `string` from the DB layer) is cast `as AllowedImageMimeType` when rebuilding `CaseFormValues` — safe because stored files are always JPEG.

**`GET /api/cases/[caseId]` and `GET /api/images/[...path]` (P2.2, committed `a849ea9`):**
- Next.js 16 route handler `context.params` is a `Promise` even for a single dynamic segment: `{ params: Promise<{ caseId: string }> }` for `[caseId]`, `{ params: Promise<{ path: string[] }> }` for the catch-all `[...path]` — must `await context.params`. Confirmed via `node_modules/next/dist/docs/01-app/03-api-reference/03-file-conventions/route.md` (version history: "params is now a promise" since v15). DI factories take `(request, context)` and integration tests build `context` as `{ params: Promise.resolve({ caseId }) }` (or `{ path: [...] }`) directly — no real Next.js router involved.
- `GET /api/cases/[caseId]` 200 body is `getCaseWithHistory(db, caseId)` returned as-is (already camelCase from the `lib/db` `toX` mappers) — full shape: `{ id, caseNumber, requestType, category, productName, purchaseDate, description, needsReview, createdAt, updatedAt, images: CaseImage[], analyses: ImageAnalysis[], decisions: Decision[], messages: ChatMessage[] }`, each array ordered oldest-first (created_at ASC). 404 → `{ error: pl.errors.caseNotFound }`. `deps = { db }` only.
- `GET /api/images/[...path]` URL shape: `/api/images/uploads/<caseId>/<seq>.jpg` (mirrors the `relativePath` `writeCaseImage` returns and `case_images.file_path` stores 1:1 — the catch-all segments are `["uploads", caseId, "<seq>.jpg"]`, joined with `Array.join("/")`, **never `path.join`** since `path.join` silently normalizes/collapses `..` segments away before the traversal check ever sees them). 200 → raw bytes + `Content-Type: image/jpeg` (hardcoded — every stored file is JPEG). Both a rejected `..` segment and a missing file return an **empty-body** `Response` (400 and 404 respectively) — deliberately no JSON error, since this endpoint's only consumer is an `<img src>`, not a JSON client. `deps = { uploadsBaseDir? }` only (no `db`).
- TypeScript gotcha: `new Response(buffer, ...)` where `buffer` is a `Buffer` fails `next build`'s type-check (`Buffer<ArrayBufferLike>` not assignable to `BodyInit`) even though `next dev`/Vitest don't complain — wrap it in `new Uint8Array(buffer)` before passing to the `Response` constructor. This only surfaces during `npm run build`'s TypeScript pass, not in tests — always run the full build before considering a route done.
- Next.js decodes dynamic route segments (including `%2e%2e` → `..`) before the handler ever sees them, so integration tests for traversal simulate this by passing already-decoded `".."` strings directly in the injected `params`, rather than trying to route a real encoded URL through Next's router.

**`GET /api/reviewer/cases` (P2.4, committed `0dd2e9f`):** no-input GET, so the DI factory `createReviewerCasesGetHandler({ db })` returns a zero-arg `GET(): Promise<Response>` (no `request` param needed — simpler than the POST factories). 200 always: `{ cases: [{ caseId, caseNumber, createdAt, requestType, category, productName }] }`, sourced from `listEscalatedCases(db)` (already filters `needs_review = 1`, newest first) and mapped from its `id` field to `caseId` — no other mapping/filtering needed since the repo layer owns the business rule. Zero escalations → `{ cases: [] }` 200, no error branch at all. Integration test seeds cases via the real `createCase`/`insertDecision` repo functions (not raw SQL) so `needs_review` is set exactly like production writes it — this is the preferred seeding pattern for any future read-route test in this project.
